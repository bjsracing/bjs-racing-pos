# Plan: Perbaiki Timezone WIB Kartu "Jam Sibuk" + Enhancement

## Konteks & Temuan (terverifikasi dari kode)
- Kartu "Jam Sibuk" di `src/pages/Dashboard.jsx` (~baris 1070) memanggil `supabase.rpc("get_peak_hours", { start_date, end_date })`.
- Frontend memetakan `r.jam` (integer 0-23) → label `${h}:00` (baris 575, 579) **tanpa konversi timezone**.
- `start_date`/`end_date` dikirim sebagai `toISOString()` → **UTC** (baris ~291, `startOfDay(subDays(...)).toISOString()`).
- RPC `get_peak_hours` **tidak ada di repo** (tidak ditemukan di `supabase/migrations/` maupun source). Definisi SQL-nya hanya ada di project Supabase (deploy).
- Gejala user: jam tampil mundur 7 jam (UTC, bukan WIB). Penyebab paling mungkin: SQL menghitung `EXTRACT(HOUR FROM created_at)` pada kolom `timestamptz` yang secara default UTC, tanpa `AT TIME ZONE 'Asia/Jakarta'`.

## Keputusan
- **Verifikasi lalu perbaiki di SQL RPC** (sesuai instruksi user: cek via Supabase API lalu tentukan). Ini paling benar & konsisten.
- NOTE: percobaan membaca RPC via `curl` Management API **diblokir oleh aturan shell** di environment ini. Verifikasi harus dilakukan lewat salah satu cara berikut (bukan dari agent ini):
  - SQL Editor di dashboard Supabase: `SELECT prosrc FROM pg_proc WHERE proname = 'get_peak_hours';`
  - atau `psql "$DATABASE_URL" -c "SELECT prosrc FROM pg_proc WHERE proname='get_peak_hours';"` (DATABASE_URL ada di `.env`).
- Setelah dipastikan SQL pakai UTC, perbaiki dengan menambahkan `AT TIME ZONE 'Asia/Jakarta'` pada ekstraksi jam.

## Task List

### 1. Verifikasi definisi RPC (wajib sebelum eksekusi)
- Jalankan query di atas (SQL Editor / psql) untuk melihat isi `get_peak_hours`.
- Konfirmasi apakah ada `AT TIME ZONE 'Asia/Jakarta'` atau `EXTRACT(HOUR FROM created_at)` mentah (UTC).
- Catat hasil: jika sudah WIB → perbaikan ada di frontend (label), jika UTC → perbaiki SQL (Task 2).

### 2. Perbaiki SQL RPC (jika UTC) — via migrasi baru
- File: `supabase/migrations/<timestamp>_fix_peak_hours_timezone.sql`.
- `CREATE OR REPLACE FUNCTION public.get_peak_hours(...)` — salin definisi existing (dari verifikasi Task 1) dan ubah baris ekstraksi jam menjadi:
  ```sql
  EXTRACT(HOUR FROM created_at AT TIME ZONE 'Asia/Jakarta')::int AS jam
  ```
  (atau bentuk setara yang dipakai di fungsi asli).
- Pastikan `GRANT EXECUTE ON FUNCTION public.get_peak_hours(...) TO authenticated;` tetap ada.
- Deploy: `supabase db push` / jalankan SQL di editor. Ini **mutasi DB** → perlu akses deploy (bukan dari plan-mode agent).

### 3. Konsistensi rentang hari (WIB) — rekomendasi, boleh masuk scope
- Karena `start_date`/`end_date` dikirim UTC, slice "hari ini"/"7 hari" dihitung dari batas UTC, sehingga data jam 00:00–07:00 WIB masuk hari sebelumnya. Untuk akurasi, ubah pengiriman parameter di `fetchDashboardData` agar menggunakan batas **WIB**:
  - Ganti `.toISOString()` pada `start_date`/`end_date` di semua `supabase.rpc(...)` di Dashboard dengan waktu yang sudah digeser +7 jam (pakai `date-fns-tz` `toZonedTime`/`formatInTimeZone` atau manual `+7h`) sebelum `toISOString()`.
  - Atau lebih simpel & konsisten: konversi di SQL-side (sudah WIB lewat `AT TIME ZONE`). Jika RPC lain (metrics, charts) juga butuh WIB, pertimbangkan helper konversi di satu tempat.
- Catatan: ini memperbaiki tidak hanya Jam Sibuk tapi semua agregat harian. Masukkan sebagai perbaikan terpisah jika user setuju (untuk menghindari perubahan terlalu luas).

### 4. Enhancement kartu "Jam Sibuk" (sesuai permintaan)
- **Highlight jam puncak:** setelah data di-load, hitung indeks jam dengan `peakValues` maksimum; tampilkan badge/subtitle "Jam Sibuk Utama: {jam}:00 WIB" + jumlah transaksi.
- **Label eksplisit WIB:** pastikan semua label jam mencantumkan "WIB" (mis. `14:00 WIB`) agar tidak ambigu.
- **Tooltip jam WIB:** pada Chart.js options, callback tooltip kembalikan `Jam {label} WIB: {value} transaksi`.
- **Warna/intensitas tetap** seperti sekarang (gradasi biru berdasar intensitas) — sudah bagus, pertahankan.
- Opsinya (pilih saat eksekusi): tambahkan anotasi teks di atas grafik "Waktu Indonesia Barat (WIB, UTC+7)".

## Failure Modes
- **Double-shift:** jika RPC sudah WIB tapi frontend juga digeser +7 → jam dobel. Oleh karena itu Task 1 (verifikasi) wajib; jangan geser label di frontend sebelum yakin RPC masih UTC.
- **Migrasi bentrok:** `CREATE OR REPLACE` harus cocok signature (parameter `start_date`, `end_date`). Ambil definisi persis dari Task 1.
- **Deploy SQL:** butuh akses Supabase (service role/token). Plan-mode agent tidak menjalankan; serahkan ke implementation agent / user.

## Rollout
1. Verifikasi RPC (Task 1).
2. Deploy migrasi SQL (Task 2) ke Supabase.
3. (Opsional) perbaiki rentang hari WIB (Task 3).
4. Enhancement UI kartu (Task 4) — murni frontend `Dashboard.jsx`.
5. `npm run build` untuk verifikasi frontend.

## Validasi
- `SELECT prosrc ...` menunjukkan `AT TIME ZONE 'Asia/Jakarta'` ada (setelah fix).
- Di Dashboard, kartu Jam Sibuk menampilkan jam sesuai WIB (mis. transaksi siang tampil di 12:00–15:00 WIB, bukan 05:00–08:00).
- Subtitle "Jam Sibuk Utama: XX:00 WIB" muncul dengan angka benar.
- Tooltip mencantumkan "WIB".
- Tidak ada pergeseran ganda (cek sekali saja di SQL).
- Build sukses.
